# OmniFocus MCP Enhanced 路线图

日期：2026-02-25

## 愿景
这个项目的长期目标，不只是继续往外暴露更多 OmniFocus API。

真正的目标是把 OmniFocus 变成一个适合和 AI 协作的任务系统，让用户可以直接说：

- “帮我规划今天。”
- “帮我清理 Inbox。”
- “把这段笔记变成一个项目。”
- “帮我看哪些项目卡住了。”
- “把这一批任务安全地重新整理一下。”

这个路线图关注的是最有杠杆的升级方向，也就是那些能让 AI 工作流真正变得自然、可靠、值得天天用的能力。

## 产品方向
我希望这个 MCP server 后续沿着 4 层能力演进：

1. 稳定执行层
- 稳定地读、建、改、移、删任务，也能做批量操作。

2. 工作流提速层
- 让用户最常见的 OmniFocus 场景更省事：每日规划、Inbox 清理、项目拆解、周回顾、重复任务维护。

3. AI 友好层
- 返回更适合大模型理解的数据和错误，让 AI 更稳、更少误操作。

4. 自动化可信层
- 批量操作更可预览、更容易部分成功、更值得信任。

## 我们真正想达到的体验
如果这个路线图走对了，用户后面应该可以非常自然地做到这些事：

- 把一段会议纪要直接变成结构清晰的项目和子任务
- 让 AI 做周回顾总结，并顺手标记 reviewed
- 一次性重组几十个任务，而不是手工一条条移动
- 直接创建和修改重复任务，不再切回 OmniFocus UI
- 先让 AI 给整理建议，再决定要不要真正执行修改

## 路线图主题

### Phase A：更安全的批量整理
这一阶段是在现有单任务移动能力的基础上，补齐高频批量重组能力。

核心内容：
- `batch_move_tasks`
- 部分成功汇总返回
- 执行前更强的校验
- 更清晰的逐项错误信息

为什么重要：
- 用户最常见的痛点之一，就是一次要整理很多任务
- 只有支持批量重组，AI 工作流才会真正省时间

### Phase B：周回顾与重复任务
这一阶段补上 OmniFocus 里非常关键、也非常适合 AI 接管的两块能力：Review 和重复任务。

核心内容：
- `repetitionRule` 支持
- `get_review_projects`
- `mark_project_reviewed`
- `set_review_interval`

为什么重要：
- 重复任务是很多用户的日常核心流程
- Review 是 OmniFocus 工作流闭环里很重要的一环

### Phase C：项目塑形与结构重组
这一阶段让 AI 不只是“加任务”，而是能真正帮用户把任务结构整理成更合理的项目。

核心内容：
- `duplicate_task`
- `convert_tasks_to_projects`
- 更顺手的项目模板化流程

为什么重要：
- 很多真实场景里，任务最初只是零散清单，后面才会发展成项目
- 如果 AI 能帮用户做结构重组，价值会比单纯新增任务高很多

### Phase D：可信执行、同步与高级元数据
这一阶段主要补可靠性和高级能力，让整个系统更接近 OmniFocus 重度用户真正想要的样子。

核心内容：
- `sync_database`
- `save_document`
- 标签组支持
- 批量操作后的更强一致性保障

为什么重要：
- 重度用户很在意大批量修改后的可控性
- 更多元数据支持，才能逐步接近 OmniFocus 新版能力

## 重点功能

### 1. `batch_move_tasks`
状态：下一阶段最重要的工作流升级

它仍然是最值得优先做的能力，因为它能直接解锁真正有用的 AI 整理场景。

API 草案：

```json
{
  "items": [
    {
      "id": "task-id-1",
      "targetProjectId": "project-id-1"
    },
    {
      "name": "任务 B",
      "targetParentTaskId": "parent-id-2"
    },
    {
      "id": "task-id-3",
      "targetInbox": true
    }
  ]
}
```

每一项包含：
- source：`id` 或 `name`
- destination：项目 / 父任务 / Inbox 三选一

期望返回模型：
- `total`
- `succeeded`
- `failed`
- `details[]`，逐项说明成功或失败原因

错误语义尽量与单任务移动保持一致：
- 目标不存在
- 名称重名（需改用 id）
- 目标选择冲突
- 非法循环转移（移到自身或后代）

首版暂不做：
- 不做位置控制（`before` / `after` / `start` / `end`）
- 不做复杂路径匹配规则

## 未来用户场景示例

### 场景 1：Inbox 清理
“把这 25 个 Inbox 任务按项目归类，剩下不明确的先留作待整理。”

需要的能力：
- 批量移动
- 更强校验
- 部分成功返回

### 场景 2：周回顾
“把需要 review 的项目列出来，帮我总结风险点，然后把看过的标记成已 review。”

需要的能力：
- Review 工作流工具
- 更适合总结的输出

### 场景 3：重复任务规划
“帮我创建一个每周五重复的行政检查清单，并确保重复规则是对的。”

需要的能力：
- `repetitionRule` 的读写支持

### 场景 4：把笔记变成结构化项目
“把这些脑暴要点变成一个项目，拆成父任务和子任务，再把最先要做的一步标记出来。”

需要的能力：
- 复制 / 转项目 / 结构重组工具
- 稳定的层级任务操作

### 接下来

### 后续
- 增加复制 / 转项目能力
- 增加同步 / 保存 / 标签组支持

## 总原则
后面的每一个新功能，都应该让这个助手更像一个真正的 OmniFocus 协作伙伴，而不只是一个“能跑脚本的接口层”。

具体来说，就是：
- 少一些脆弱的多步提示词
- 多一些完整工作流支持
- 少一些大批量操作的不确定性
- 多一些更自然的规划与回顾体验
